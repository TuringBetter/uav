# Raft 控制指令共识测试

## 一、测试场景

**场景名称：** 5 架无人机控制指令强一致性下发

**场景描述：**  
模拟 5 架无人机编队通过 Raft 算法选举出唯一 Leader。然后由 Leader 连续发布多条航点变更指令 (Waypoints)。通过 Raft 日志复制确保所有节点最终获得一致的控制指令序列，且被成功 Commit。测试目标是验证强一致性算法的可用性并衡量其通信开销。

```
        Node 2        Node 3
          ╲          ╱
            Node 1 (👑 Leader)
          ╱          ╲
        Node 4        Node 5
```
**拓扑：** 全连接 (Full Mesh)，5 个节点互为 Peer。

## 三、初始状态与目标状态

### 初始状态
全部节点均为 Follower 状态，日志 `log` 长度为 0，`commitIndex` 为 0。

### 目标状态（收敛）
在 Leader (如 Node 1) 收到 10 条外部提案后：
1. **Leader 状态：** `role = leader`, `logLen = 10`, `commitIndex = 10`
2. **Follower 状态：** `role = follower`, `logLen = 10`, `commitIndex = 10`

收敛判定条件：
```
所有节点的 status.LogLen == 10 
所有节点的 status.CommitIndex == 10
```

## 六、测试发现与代码修复

在开发此测试框架的过程中，捕获了 Raft 算法原本存在的两个缺陷并已完成修复：

1. **并发时生成完全相同随机数 (Split Vote)**
   原代码使用 `time.Now().UnixNano()` 作随机种子，导致同一机器上极高并发初始化的数个无人机节点产生完全相同的竞选超时器从而“永恒平票”。通过在 Seed 中按 `node.ID()` 大幅偏移混淆修复。
2. **幽灵选票回复 Bug**
   旧实现里，Follower 节点回复 `RequestVoteReply` 包含的 `Term` 是其尚未经过检查而 `StepDown()` 之前的旧的本地 `Term`（比如为0）。这导致 Candidate 在进行最新纪元的选票核算时拒绝该选票。通过后置初始化 `reply.Term` 解决，集群目前能在百毫秒内迅速推举出唯一 Leader。

## 七、与 Gossip 的开销对比建议

此版本的 Raft（10指令下发）产出了约 `16 KB`，而此前的 Gossip 实验(5状态同步)产出约为 `8.2 KB`。  
由于 Raft 的 Leader 需要高频(`50ms/回`)维护广播心跳和状态，如果无人机编队只是弱状态同步，**Gossip 的开销将显著更低且容忍断连更优**。  
而当涉及到需要**绝对一致**的任务(如更换关键航点、控制攻击)，Raft 测试也证明了在强网络下可以带来极高的延迟吞吐(150ms收敛10条指令)。后续建议使用 `tc` 切断 Leader 并验证重新选主时间。
